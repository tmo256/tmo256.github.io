Introduction

The goal of this document is to facilitate the creation of a common UI / UX experience for all UL external-facing web applications; it provides a set of common rules that can be applied to all existing and new applications and features. It is not intended to be comprehensive or to answer every possible question that might arise in developing an application. Designers and devs are encouraged to apply common sense and good judgement!

By providing goals, citation and context wherever possible, this document seeks to explain the reasoning behind each suggested rule, which by extension should help application producers understand when and why to disregard them. However, items that are highlighted in teal should generally be adhered to unless there are very strong reasons to disregard.

This document is intended to be technology agnostic and provide common guidelines whether the underlying implementation is SASS or LESS, Bootstrap or Semantic UI, React or Angular, etc.

Common Foundation

 Implementing the common foundation in an application will accomplish 80% of a common look and should (and is intended to) be able to be accomplished quickly. 

·Header color and UL logo. UL red header is preferred but for historical / persona reasons other choices are acceptable- Good Guide, for example, serves an entirely different persona from what is more typical for other UL apps.

·Footer color. Dark grey, common link styles and elements (copyright, etc.) where possible.

·Font: Adopt Open Sans for headers at minimum. 

·Icons: font awesome (there is some cost associated with this).

·Button styles: Padding / border radius, shape, border, shadow, mouseovers (transitions, relative lightness/brightness). Not necessary, color or typography.

Style Guide

Web Standards

The principles and code fragments below assume modern browsers, which includes the latest (desktop and mobile) versions of Chrome, Safari and Firefox, as well as IE 11+. Target MS Edge only where your user adoption makes it relevant, as Microsoft plans to phase out support for this browser. Support of other, older browser versions may be required where audience usage warrants, which may lead to the need to disregard some style guidelines.

Prefer graceful CSS fallbacks to polyfills where possible and employ tools such as Modernizr to help detect and create these fallback styles. When writing JavaScript, use ECMA6 syntax with caution, as some features of the new script version have been adopted more widely than others.

If in doubt, Can I Use in combination with your audience analytics can provide guidance on what browser features are reasonable to employ for your application. Post-processor tools like PostCSS can be integrated into automated builds to support next gen CSS functions and prefixed styles (such as prefixed flex rules to support flex in IE 10) to stretch limitations.

In general, prefer

·HTML5.

·CSS3 and new CSS modules with adequate browser adoption.

·ECMA5 and ECMA6 features with adequate adoption (or lightweight polyfills).

Internationalization (i18n)

UL is a global company operating in many international markets. Consequently, designers and developers should plan for localization, internationalization and translations. If your application has been designed responsively, you will most likely be able to accommodate the variations of text length resulting from these processes.

For proper text display:

·add the lang attribute to your root html tag. Prefer html to body to accommodate localized meta tags.

·include a meta tag with a charset of UTF-8 or UTF-16, depending on the language you’re displaying.

Without preparation, i18n can affect any HTML element. Look to accommodate for text wrap and horizontal overflow. Pay particular attention to:

·     navigation and menus.

·     tab labels.

·     button text.

·     table column headers.

·     any element content with specified heights.

In general, assume that text will be longer in German, Spanish and French, and much shorter in Asiatic languages.

Also consider ISO formatting in form input fields. Be sure to accommodate a variety of date, price, zip/province code and telephone formats.

Common Terminology

Use common language for common features both within and between different applications.

·Sign in to refer to the process of authenticating to gain access to the application.

·Sign out to refer to removing the authenticated state and leaving the application.

·Account to refer to user profile settings.

Typography

UL has standardized on the Open Sans font family for web sites and applications. If Open Sans is unavailable, use Arial as a substitute.

In order to create a visual hierarchy and provide visual contrast and rest for the eyes, use Segoe and Helvetica Neue as the base body copy; these fonts will be available on the vast majority of operating systems and offer a variety of font weights and variants.

For accessibility, set body font size to 100%, which respects individual users’ text preferences. Browsers typically default font size to 16px, so the majority of users will experience the site as envisaged, while those with disabilities will have their preferences respected.

The typography specifics below are based on the concept of typographic modular ratios. The CSS included was created using the Major Third scale (which is similar to the UL marketing typography). NOTE: For more information on the concept of a typographic modular scale, see More Meaningful Typography. For the scale calculator, see type-scale.com.

html {font-size: 100%}

/* as a result of this rule, in most browsers 1rem = 16px */

body { 

  font-family: "Segoe UI","Helvetica Neue",sans-serif;

  font-weight: 400;

  line-height: 1.45; 

  color: #333;

}

p {margin-bottom: 1.25em;}

h1, h2, h3, h4, h5 { 

  margin: 2.75rem 0 1rem;

  font-family: 'Open Sans', sans-serif;

  font-weight: 300; /*correlates to Open Sans Light*/  

  line-height: 1.15;

}

h1 {

  margin-top: 0;

  font-size: 3.052rem;

}

h2 {font-size: 2.441rem;}

h3 {font-size: 1.953rem;}

h4 {font-size: 1.563rem;}

h5 {font-size: 1.25rem;}

small, .text_small {font-size: 0.8em;}

Note: Header font sizes are listed in rems to avoid unintended outcomes in nested elements, while .small is defined in ems in order to be relative to the size of the parent element. Carefully consider the intended consequences of nested elements when defining styles in ems and rems.

Colors

UL’s primary brand colors are as follows:

·UL Red: #C02032

·UL Black: #000000

·UL White: #FFFFFF

These colors should be preferred as described in the Common Foundation section. Other approved UL colors can be found on the brandhub site maintained by the corporate marketing team.

Accessibility

Note, in certain circumstances UL’s red brand color can present challenges to users with color blindness. At a minimum,

·avoid combining red and green text/backgrounds/juxtapositions;

·when using color to convey meaning (such as red text for error messages, green for success, etc.), also use symbols and words to convey that meaning.

In general, all text should be readable by measuring a contrast ratio of at least 4.5:1 (as low as 3.0 is acceptable in some circumstances) between text and background. A number of contrast checker online tools are available; Chrome developer tools also provides a tool to do this (tutorial).  Best practice is to make sure that the information is conveyed other than just by color.

Icons

Icons should be used as accents to support the topic and ideas being presented. As noted above, a combination of color, text and icons will convey the message best.

Use Font Awesome wherever possible for icons. Use the Light version, available to Pro users, when possible.

Font Awesome icons can be rendered as a vector image in any size of browser and don’t require any extra downloads or server resources; the library includes a large number of icons covering a wide gamut of objects.

On occasion, you may need to develop custom icons for special cases. Implement these icons in the code as SVGs, which allows for smooth and responsive resizing.

Responsiveness

Sites and web applications should be responsive if at all possible. A common set of break points will provide a consistent experience for those that switch between devices on a mobile device.

·576px max – Small

·768px max – Medium

·992px max – Large

·1200px max – Extra large

·1600px max – Double extra large

Prefer fluid layouts to fixed width if the content will support it. When displaying paragraphs of text in fluid layouts, you may need to limit (via media queries) container widths for readability.  Conventionally, this will be 34 to 36ems, but use good judgement.

See "Components" section below for specific responsive-related guidelines.

Prefer a top bar navigation with the product name on the left and the UL logo as the last element to the right, along with whatever other minimal navigation elements are required between the logo and the product name. Accessibility

In addition to the accessibility guidelines specified in other sections, employ general best practices when coding to accommodate ADA and general accessibility. Employing these elements will not only benefit the user, it will improve SEO on your apps. The WCAG, a W3 working group, offers extensive tutorials on accessibility.

ARIA attributes play a significant role in achieving accessibility.

ARIA (WAI-ARIA if you want to be formal) is a set of attributes that you can add to HTML elements.

These attributes communicate role, state and property semantics to assistive technologies via the accessibility APIs implemented in browsers. 

Coding General HTML Elements

Accessibility should be implemented by conforming to semantic HTML practices, as well as using ARIA attributes/labels. See individual component and layout sections for suggested implementation of semantic HTML.

Using ARIA is particularly important in forms and inputs, while "role" attributes are particularly useful in elements (like buttons) that perform crucial functions but do not semantically indicate their uses. Refer to the WCAG documentation for use in specific HTML elements and common components (such as tooltips).

Images

Google’s Lighthouse and Page Speed tools take special interest in image accessibility. Be sure to include alt text on all img tags, as its absence is detrimental to your audit results and SEO.  The alt text should description should provide the same level of understanding that a visual user would have. Where at all possible, do not use images for buttons, navigation or significant text. Minimize text in images in general; if text is present in images, use that text as the alt value.

Component Library

This section contains common components used by web applications with a rational for when and why they should be used. It is not intended to be comprehensive.

Alerts

Use an alert to implement notices of success or failure for a persistence action (i.e. form submit). On a full-page view, these notices should fall in a consistent position from page to page; for example, directly beneath the main page title or the page lead text. If an alert message appears within a modal window, the alert should be prepended as the first element the modal content, in general (though specific layouts may require a modified implementation).

Where the alert is a direct result of an action by the user, the alert should have a dismissible option (close button). In other circumstances, alerts may be used to provide help text or other notices, and these elements should not include a close button.

Accessibility

·For a time-sensitive alert, use ARIA role='alert'.

·For an alert that is not time-sensitive but that contains an important prompt or piece of information for the user, use ARIA role='alertdialog'. (The definition of the ARIA role='dialog' is: "an application window that is designed to interrupt the current processing of an application in order to prompt the user to enter information or require a response").

·Alerts should not be visually hidden if they are not being used. If you do not want a screen reader to pick up an alert, use the aria-hidden='true' HTML attribute and/or display: none in your CSS.

Anchors & Buttons

The guidelines below describe usage and styling of button and anchor elements. See "Forms" for specific uses within the context of a form.

Buttons: Coding

·Use a button element to trigger any JavaScript function that does not use a URL. Basically, instead of using an a tag with an href="Javascript:void(0)", or href="#", use a button element. If the user can open a reference via a right-click or 3D touch, use an anchor.

This might include:

onon-AJAX modal triggers.

o popover triggers.

o tooltip elements.

o dismissing buttons in modals and alerts.

o opening and closing/expanding hidden text.

·In many of the cases (such as some listed above), the button may appear to the user to be an inline link. Use CSS to modify the button to look like a link.

·Always use the type="button" attribute-value to ensure a button does not inadvertently act as a form submit. Safari, Chrome and other browsers treat button elements as submit tags when nested in forms.

Buttons: Styling

As noted, button elements may be styled to appear as either buttons or as links, depending on their conceptual use.

Style elements as a button when:

·indicating persistence or as a step in a process leading to persistence. This includes actions such as save, save and move forward or back ("next" and "previous" step), submit, delete, update.

·creating call to action (CTA). The number of CTAs should be limited on a view and should strongly indicate to the user what their next action should be.

Unless noted elsewhere in this document, style the button as a link in most other situations.

Buttons: Responsiveness

In most circumstances, elements styled as buttons should become block elements in medium and smaller viewports in order to accommodate touch behavior; the exception to this rule might be for "Next" and "Previous" buttons. Be sure to create adequate spacing and padding to allow all users to use your buttons on a touchscreen.

Buttons: Usability

·Use sentence case for button labels.

·Limit characters. Button text should be as short as possible with action words that clearly explain what the button does (Create Account, Save, Search).

·Be consistent in this terminology across the application, such as "Save" vs "Submit".

·Lead with a verb where possible. For example, "Create Account" not "Account Creation".

·Icons can also be helpful. At times, consider adding an icon to signal specific actions (Download, Open in a new window, etc.).

·Use consistent class variations to distinguish types and convey meaning. For example, "Next" and "Previous" button steps should use a default or plain button class, while a submit or CTA button would implement a special primary.

Anchors: Coding

·Whenever an element uses a URL or in-page navigation, use an anchor tag. This includes

ostandard inline links.

ojump nav links (href=#element-id).

oAJAX modal triggers, where the user can also open content in a new tab (by right-click or 3D touch).

oall cases described in "Forms".

owhen in doubt, prefer an anchor tag.

·As noted above, a trigger for a modal may appear either as a link or a button, depending on circumstances. If the request is made using the URL in the href attribute, use an anchor.

·When using an anchor tag as a button, include the role="button" attribute.

Anchors: Styling

Inline anchor style should use standard body font family and size, line height, etc, and, by default, an underline (text-decoration) to indicate a link to the user. This allows the designer to use colored text as a design element in headers or banners without creating confusion for the user; users should be able to identify links by more than color alone.

In some cases, the context of the link will provide the user with the indication of clickability; for instance, within a standard navigation in the header, or within a drop-down submenu. In those cases, underlines or other indicators may not be required.

Where an interactive link reveals (expands or creates) text, include an indicator of further content, such as an angle or caret icon. Ideally, this indicator would change orientation (animated) to indicate open and closed.

When designing for touch display, make sure to provide adequate space around every link, just as with buttons. Pay specific attention to lists and menus.

Input/Submits

Use styling described in the "Buttons: Styling" and "Buttons: Usability" sections.

·Where possible, submit a form using input type="submit".

·As noted in "Forms", attach any validation or pre-processing to the submit event for forms rather than click events on these elements. This will enable the use of the submit button, as well as ensure that submission by using the "Return" key or by assistive technology software will always execute the required JavaScript.

Other Cases

In addition to the cases noted above (primarily buttons that trigger JavaScript functions), buttons may be styled as links or vice versa.

·All elements in an "Action" table cell should appear to the user to be links. The actions may be a mix of regular anchor tags, such as an anchor link to "Edit;" buttons, such as a confirmation modal triggered for a delete; or a submit button, as when submitting a form to make a saved search an alert.

·Regular anchors may be styled as buttons for an extraordinary or analogous call to action, such as "Registration" links.

·Where a mix of form submit and anchor elements should be viewed as siblings of equal importance, be consistent in displaying each element as either buttons or links.

Confirmation Dialogs

Use a confirmation dialog to confirm a process or persistence that cannot be easily undone or recreated; or, to confirm data persistence where the user may not be aware that persistence is about to occur (form inputs are not visible, for example). Most commonly, use this type of dialog when deleting or removing items. The dialog should interrupt the process to the point that the action cannot complete without explicit input (e.g., button click) from the user.

Buttons and Links (Again)

Forms commonly require actions other than a simple submit. In some cases, these actions should be performed by buttons and sometimes by links.

Note that although the rules below refer to buttons and links, the elements need not specifically be input, button or anchor tags; rather, we refer to the styling of the element (see "Anchors and Buttons" for more information on how to style these elements). For the vast majority of instances in the context of forms, however, links should be anchor tags and submit buttons input elements of type "submit" (see "Anchors and Buttons" for details on when to use each type of element).

Use the general guideline of the necessity of a URL to differentiate between using an anchor or a button.

Buttons

·Submit (Create/Update)

·Delete

·Reset*

·Any action that initiates a data transaction

Links

·Cancel*

·Forgot Password

·Actions within tables (e.g. "Remove" link in a list, which performs a form submit)* . Note this represents a case where the element appears to the user to be a link but should actually be a button.

·Any other action that does not initiate a data transaction

*Note: "Reset" is a native action that resets the data in an HTML form to its state upon page load; once clicked, the user will stay on the current page. "Cancel" will abandon the form by sending the user to another page, removing the form element, et al. Where sensible, prefer Cancel to Reset, and both may be used. For example, a calculator form would require a reset, where a new item create form is better served by cancel.

Submit Events

As discussed above, although buttons and links do not necessarily need to correspond to their native HTML element tags, forms should generally be executed with input elements of type "submit." Use a submit input instead of input of type="button". (See "Anchors and Buttons" for more information on when to use alternative element types.)

Any JavaScript in need of execution upon form submission should be attached to the form’s submit event, rather than attached to a click event of a button or link. Additionally, users will be able to submit forms by using the "Return/Enter" key, within a touch device keyboard, or by other methods (especially those used by accessibility browsers); attaching events to form submits circumvents all these issues.

JavaScript Validations

In most cases, JavaScript validations are not necessary. Some situations may make client-side validation more useful to the user, however.

Use client-side validations when:

·the form is submitted via Ajax. Generally, this will make action flow more fluidly for the user.

·the form contains a file upload input. Because of security issues, uploaded files are detached on page load, which makes round-trip validation poor UX (as the user is forced to reattach the file).
(Note that some browsers expose more attributes of uploaded files than others, making specific validations relating to file size, type, etc., inconsistent across browsers.)

·the form contains typeahead inputs that might be lost upon reload.

Best Practices

·Implement the same error styling to the elements as the user would see from a server-side validation.

·Validate on form on submit rather than on input. However, a multistep input, such as typeahead field, may be validated on value select for better interaction flow.

·Clear errors on re-validation. Depending on the input type, as in the case of password field, clearing the error on field change may be more helpful to the user.

Success & Error Handling

As described in the "Alert" section, indicate success by including a visual alert in the resulting view. This alert should be dismissible (it is not necessary to programmatically close the alert after a set period of time).

When a view includes a form with error fields, an alert should be visible at the top of the view with a generic error message, such as "Please correct the input errors." Do not list individual field errors in this message. Specify these errors by styling each invalid input.

Images

Responsive Images

HTML5 provides srcset, an attribute designed to deliver the optimal file for the viewport size of each specific device requesting an image. When specified, the browser determines the smallest (file size) image to download based on its resolution and element’s dimensions.

<img src="small.jpg" srcset="medium.jpg 1000w, large.jpg 2000w" alt="yah">

For details on how to use srcset to deliver responsive images, see this article. Note that IE 11 does not support the srcset attribute, so you may wish to use the desktop size version of the image as the src attribute value where your audience browser usage indicates.

When displaying images, it’s important to remember the largest version should be formatted for viewing across all viewport sizes. To ensure proper display, give the image a max-width property of 100%. However, in some instances, including the max-width may not be enough. While it is always important to test views in multiple browser sizes, images may require more specific adjustments to responsive layouts than any other page element. In many frameworks, layout becomes liquid at specific viewport sizes (i.e. at < 768px all sizing is based on percentage rather than specific pixel sizes). With a broad variety of sizes for small touch devices, be sure to test images thoroughly.

Image Sizing and Resolution

Images should display at the correct resolution for all types of screen resolutions. Historically, images delivered to screens have required resolution of no more than 72ppi. However, with the advent of Retina and other high-resolution screens, resolutions have increased to 144+ppi. Unfortunately, most web file formats can only be saved as 72ppi and do not display crisply on high resolution screens in their native size.

To address this problem, you will need to create at least one version of the image at twice the maximum size it will appear within the site, and then use CSS styling to adjust dimensions to half its native size. For instance, if you have an image that needs to be displayed at 100×50, then you should save the image at 200×100 and use height and width properties in display.

It is important to recognize that the largest display of an image size may not correspond to the largest media screen size. This means that an image may actually display larger in mobile or tablet view than in desktop, depending on the page layout at each responsive break. Again, responsive testing is vital in ensuring the image displays crisply and responsively in all formats.

Using srcset to deliver responsive images means you should determine the largest display size as outlined above, and then make smaller versions of this image as well. Use the devices your analytics indicate to determine your target sizes or use an online tool, such as the Responsive Image Breakpoints Generator, to create a subset of images.

Image File Formats

Because of high resolution proliferation, PNG formats will best serve your web collateral. Consider indexed or 8-bit PNG format for illustrative images, rather than the full 24-bit resolution. Much like indexed GIF images, this lower bit-per-pixel display will have a considerably smaller file size. Larger photographic images will be better saved as JPGs to reduce delivery size. As noted below, always use a tool to optimize these images for maximum compression, as client-side auditors will penalize even the most diligently saved file.

Save vector graphics as native SVG files. For instance, your application logo, which may use a custom font and be displayed in a variety of sizes and colors, could be implemented as an SVG image. If you believe an image may be suited for this type of display, also consider the cons, including rendering time and processing. The object implementation of SVG files with external CSS is the optimal implementation. For more information, see css-tricks.com.

Image Optimization

As noted above, non-optimized, uncompressed images can be quite large in terms of file size and this can have an outsize effect in high network latency situations like cellular networks for mobile devices or even just browsers that are a long way from your data center. I.e. in China.

It is critical to use proper tools and techniques to fully optimize image formats to reduce file size and then at as high a rate of lossy compression as is tolerable while still maintaining required image quality. While it is possible to accomplish a good file size optimization in your image editor, there are many freely available tools on the web that make this easy; again, even highly optimized images may trip a penalty from auditing tools. Online image processors for optimization include:

·Optimizilla at https://imagecompressor.com makes fine-tuning images exceptionally easy.

·Google Page Speed offers downloadable files upon audit. https://developers.google.com/speed/pagespeed/insights/

There are also ways to integrate image compression and optimization in version control systems so that it happens automatically on any new image that is introduced to the repo. Cloudflare also provides a solution for optimizing image delivery.

Modals

The main purpose of modals is to allow for the completion of a secondary or sub-task without abandoning the initial or primary task completion. For instance, when a user performs a keyword or phrase search, they may save the search for future use. In order to maintain the current list of results, the save process occurs within a modal. Sub and secondary tasks performed in modal should comprise a limited number of steps and options, so not as to divert from the primary task completion.

You may also use modals to alert users where JavaScript or AJAX transactions have processed in the background. For example, imagine a registration form where a number of fields may be completed. Several validations may occur asynchronously on the email input; when errors or potential problems occur, a modal can be used to catch the attention of the user who have moved away from the field. In this case, the modal alerts the user to important information, interrupting their current task without entirely removing them from it.

Finally, a modal may serve as a dialog in circumstances described in the "Confirmation Dialog" section.

Accessibility

·If possible, modals should be able to be used by the keyboard alone.

Tables

Tables should be used to display easily scanned and/or comparable data.

Responsive Tables

Where tables are used in responsive design, ensure display in medium and small browser views by employing one or a combination of these methods:

·Using data-labels attributes in table cell tags, combined with media query CSS to transform each cell into a stacked row. Many versions and tutorials on this method are available, including this one by Matt Smith. (See link for a visual example of this method.)

oPros: This method is widely supported and accommodates a lot of data legibly.

oCons: By stacking the data, comparing cells becomes more difficult.

·Suppress less important columns where possible, allowing the remaining cells to fit comfortably in portrait mode on common mobile devices.

oPros: Supports the original goal of the table, keeping the contents scannable and comparable.

oCons: May eliminate useful information for the user. (In general, tables should not contain much information that is not necessary for the user. However, the circumstances under which the user uses a mobile device may change which pieces of information are more important.)

·Add horizontal scrolling to the table’s parent object, while including some visual indicator that more content is available beyond the current view screen. Indicator might be a visible horizonal scrollbar or a gradient overlay.

oPros: Requires less coding and maintenance while displaying all the table data.

oCons: Horizontal scrolling is not always touch-device friendly; users have a hard time maintaining a connection between identifying cells. Creating the visual indicator of "more" can also be unclear.

Sorting

Prefer sorting by:

·select/dropdown element.

·table header ascending/descending.

When tables are responsive and your audience may access your application using multiple devices, prefer sorting via dropdown rather than using table headers to maintain a consistent experience. This allows for sorting separate from the table itself, making all responsive options viable.

Accessibility

·Header cells should utilize the TH tag

oAll header cells should contain text. 

·Data cells should utilize the TD tag

 

Tabs

Use tabs to break up related information within the same context. However, using tabs can present multiple challenges in responsive situations, especially in mobile-sized views. These problems include inadequate horizontal space to display tab labels (especially when translated into multiple languages, where text width will be unknown), fluctuating content element heights (affecting layout as the tab changes), difficulty with internal scrolling of content, etc. Avoid where possible.  

If tabs are necessary and present problems in smaller responsive views, display all content. If the content is exceptionally long, maintain tab links as a menu of jump links and include a "Return to top" link back to the submenu.

Tooltips & Popovers

Meta information, such as help text and input examples, may be supplied to the user via tooltips and popovers. Due to responsive requirements, using popovers poses logistical difficulties, but may be used judiciously. Length of text is the primary consideration for usage, but function also plays a role.

·Use tooltips for short textual information attached to specific inline elements, like help for field inputs. Tooltips should contain no more than 75 characters.

·Use a popover when providing stand-alone information, such as explanatory text or extra information of indeterminate length. For example, help that applies to an entire form (as opposed to a single element) could be contained in a popover. Popovers may also be preferred to dialogs in certain circumstances, such as upsell messages for page features.

Generally, tooltips and popover elements should behave in the same way:

·The trigger element that opens the element should also close it. That means that all triggers toggle to open and close (per guideline specifications, elements that perform no function other than that of a toggle should always be buttons, though those buttons may be styled to appear as links).

·In most cases, prefer triggers that open the component explicitly, rather than supplementally. For instance, do not automatically open a tooltip when a user focuses on an input field; provide a separate link to open the element.

·Because of touch requirements, trigger both components with click events rather than hover or mouseover. This will also aid accessibility and usability concerns by avoiding unintended display of information (inadvertent mouseovers).

·Opening a new popover or tooltip does not necessarily require closing other open elements of the same type. Generally, open tooltips should probably be closed, while open popovers should remain open.

·Popovers may also include a separate close button in the title bar.

Accessibility

Include ARIA attributes in the trigger and display elements.

·The element that serves as the popover/tooltip container has role=tooltip.

·The element that triggers the popover/tooltip references the popover/tooltip element with aria-describedby.

Style Guide

Web Standards

The principles and code fragments within this document assume modern browsers, which includes the latest (desktop and mobile) versions of Chrome, Safari and Firefox, as well as IE 11+. Target MS Edge only where your user adoption makes it relevant, as Microsoft plans to phase out support for this browser (as of fall 2019, early 2020 is the estimated release date of MS Edge built on Chromium). Support of other, older browser versions may be required where audience usage warrants, which may lead to the need to disregard some style guidelines.

Prefer graceful CSS fallbacks to polyfills where possible and employ tools such as Modernizr to help detect and create these fallback styles. When writing JavaScript, use ECMA6 syntax with caution, as some features of the new script version have been adopted more widely than others.

If in doubt, Can I Use in combination with your audience analytics can provide guidance on which front-end features and technologies are reasonable to employ in your application. Post-processor tools like PostCSS can be integrated into automated builds to support next gen CSS functions and prefixed styles (such as prefixed flex rules to support flex in IE 10) to stretch limitations.

In general, prefer:

  • HTML5
  • CSS3 and new CSS modules with adequate browser adoption.
  • ECMA5 and ECMA6 features with adequate adoption (or lightweight polyfills).

Internationalization (i18n)

UL is a global company operating in many international markets. Consequently, designers and developers should plan for localization, internationalization and translations. If your application has been designed responsively, you will most likely be able to accommodate the variations of text length resulting from these processes.

For proper text display:

  • add the lang attribute to your root html tag. Prefer html to body to accommodate localized meta tags.
  • include a meta tag with a charset of UTF-8 or UTF-16, depending on the language you're displaying.

Without preparation, i18n may affect the display of any HTML element. Look to accommodate for text wrap and horizontal overflow. Pay particular attention to:

  • navigation and menus.
  • tab labels.
  • button text.
  • table column headers.
  • any element content with specified heights.

In general, assume that text will be longer in German, Spanish and French, and much shorter in Asiatic languages.

Also consider ISO formatting in form input fields. Be sure to accommodate a variety of number, date, price, zip/province code and telephone formats.

Common Terminology

Use common language for common features both within and between different applications.

  • "Sign in" to refer to the process of authenticating to gain access to the application. (This may necessitate changing existing labels or text from "Log in" and "Login".)
  • "Sign out" to refer to removing the authenticated state and leaving the application. (As above, change "Log out," etc.)
  • "Account" to refer to user profile settings.

Typography

UL has standardized the Open Sans font family for web sites and applications. If Open Sans is unavailable, use Arial as a substitute. To include Open Sans from the Google Font repository, add this link to your document header:

<link href="https://fonts.googleapis.com/css?family=Open+Sans:400,600&display=swap" rel="stylesheet">

This example includes the font at 400 (normal) and 600 (semibold) weight. Find out more about this font and how to use it at Google Fonts.

In order to create a visual hierarchy and provide visual contrast and rest for the eyes, use Segoe and Helvetica Neue as the base body copy; these fonts will be available on the vast majority of operating systems and offer a variety of font weights and variants.

For accessibility, set body font size to 100%, which respects individual users' text preferences. Browsers typically default font size to 16 pixels; when your body copy is set to 100%, 1rem = 16px. If your design calls for 14px type as the base, you can specify individual elements (or "*") to .875rem. The majority of users will experience the site as envisaged. However, users who have set their default browser font size to 20px will have their preferences respected; those users will still see a larger font size, as .875 of 20 displays 17.5px.

The typography specifics below are based on the concept of typographic modular ratios. The CSS example was created using the Major Third scale (which is similar to the UL marketing typography). NOTE: For more information on the concept of a typographic modular scale, see More Meaningful Typography. For the scale calculator, see type-scale.com.

html {font-size: 100%}

/* as a result of this rule, in most browsers 1rem = 16px */
body { 
  color: #333;
  font-family: "Segoe UI","Helvetica Neue",sans-serif;
  font-weight: 400;
  line-height: 1.45; 
}

p {margin-bottom: 1.25em;}

h1, h2, h3, h4, h5 { 
  font-family: 'Open Sans', sans-serif;
  font-weight: 300; /*correlates to Open Sans Light*/  
  line-height: 1.15;
  margin: 2.75rem 0 1rem;
}

h1 {
  font-size: 3.052rem;
  margin-top: 0;
}

h2 {font-size: 2.441rem;}

h3 {font-size: 1.953rem;}

h4 {font-size: 1.563rem;}

h5 {font-size: 1.25rem;}

small, .text_small {font-size: 0.8em;}

h1.Heading = 3.052rem

h2.Heading = 2.441rem

h3.Heading = 1.953rem

h4.Heading = 1.563rem

h5.Heading = 1.25rem
Small Text

Note: Header font sizes are listed in rems to avoid unintended outcomes in nested elements, while .small is defined in ems in order to display relative to the size of the parent element. Carefully consider context and unintended consequences of nested elements when defining styles in ems and rems.

Colors

UL's primary brand colors are as follows:

  • UL Red: #C02032
  • UL Black: #000000
  • UL White: #FFFFFF

These colors should be preferred as described in the Common Foundation section. Other approved UL colors can be found on the brandhub site maintained by the corporate marketing team.

Accessibility

In certain circumstances, UL's red brand color can present challenges to users with color blindness. At a minimum,

  • avoid combining red and green text/backgrounds/juxtapositions;
  • when using color to convey meaning (such as red text for error messages, green for success, etc.), also use symbols and words to convey that meaning.

In general, all text should be readable by measuring a contrast ratio of at least 4.5:1 (as low as 3.0 is acceptable in some circumstances, where font weight or text size are sufficiently large) between text and background. A number of online contrast checker tools are available; Chrome developer tools also provides a tool to do this (tutorial).

Icons

Icons should be used as accents to support the topic and ideas being presented. As noted above, a combination of color, text and icons will convey the message best.

Use Font Awesome wherever possible for icons. Use the Light version, available to Pro users, when possible.

Font Awesome icons can be rendered as vectors at any size and don’t require any extra downloads or server resources; the library includes a large number of icons covering a wide gamut of objects.

On occasion, you may need to develop custom icons for special cases. Implement these icons in the code as SVGs, which allows for smooth and responsive resizing.

Responsiveness

Sites and web applications should support responsive layouts if at all possible. Using common break points will provide a consistent experience for those that switch between applications on desktop, tablet and mobile devices.

  • 576px max – Small
  • 768px max – Medium
  • 992px max – Large
  • 1200px max – Extra large
  • 1600px max – Double extra large

Prefer fluid layouts to fixed width if the content will support it. When displaying paragraphs of text in fluid layouts, you may need to limit (via media queries) container widths for readability. Conventionally, this will be 34 to 36ems, but use good judgement. See "Components" section below for specific responsive-related guidelines.

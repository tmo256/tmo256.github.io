Style Guide

Web Standards

The principles and code fragments below assume modern browsers, which includes the latest (desktop and mobile) versions of Chrome, Safari and Firefox, as well as IE 11+. Target MS Edge only where your user adoption makes it relevant, as Microsoft plans to phase out support for this browser. Support of other, older browser versions may be required where audience usage warrants, which may lead to the need to disregard some style guidelines.

Prefer graceful CSS fallbacks to polyfills where possible and employ tools such as Modernizr to help detect and create these fallback styles. When writing JavaScript, use ECMA6 syntax with caution, as some features of the new script version have been adopted more widely than others.

If in doubt, Can I Use in combination with your audience analytics can provide guidance on what browser features are reasonable to employ for your application. Post-processor tools like PostCSS can be integrated into automated builds to support next gen CSS functions and prefixed styles (such as prefixed flex rules to support flex in IE 10) to stretch limitations.

In general, prefer:

HTML5

CSS3 and new CSS modules with adequate browser adoption.

ECMA5 and ECMA6 features with adequate adoption (or lightweight polyfills).

Internationalization (i18n)

UL is a global company operating in many international markets. Consequently, designers and developers should plan for localization, internationalization and translations. If your application has been designed responsively, you will most likely be able to accommodate the variations of text length resulting from these processes.

For proper text display:

add the lang attribute to your root html tag. Prefer html to body to accommodate localized meta tags.

include a meta tag with a charset of UTF-8 or UTF-16, depending on the language you’re displaying.

Without preparation, i18n can affect any HTML element. Look to accommodate for text wrap and horizontal overflow. Pay particular attention to:

navigation and menus.

tab labels.

button text.

table column headers.

any element content with specified heights.

In general, assume that text will be longer in German, Spanish and French, and much shorter in Asiatic languages.

Also consider ISO formatting in form input fields. Be sure to accommodate a variety of date, price, zip/province code and telephone formats.

Common Terminology

Use common language for common features both within and between different applications.

Sign in to refer to the process of authenticating to gain access to the application.

Sign out to refer to removing the authenticated state and leaving the application.

Account to refer to user profile settings.

Typography

UL has standardized on the Open Sans font family for web sites and applications. If Open Sans is unavailable, use Arial as a substitute.

In order to create a visual hierarchy and provide visual contrast and rest for the eyes, use Segoe and Helvetica Neue as the base body copy; these fonts will be available on the vast majority of operating systems and offer a variety of font weights and variants.

For accessibility, set body font size to 100%, which respects individual users’ text preferences. Browsers typically default font size to 16px, so the majority of users will experience the site as envisaged, while those with disabilities will have their preferences respected.

The typography specifics below are based on the concept of typographic modular ratios. The CSS included was created using the Major Third scale (which is similar to the UL marketing typography). NOTE: For more information on the concept of a typographic modular scale, see More Meaningful Typography. For the scale calculator, see type-scale.com.

html {font-size: 100%}

/* as a result of this rule, in most browsers 1rem = 16px */

body { 

  font-family: "Segoe UI","Helvetica Neue",sans-serif;

  font-weight: 400;

  line-height: 1.45; 

  color: #333;

}

p {margin-bottom: 1.25em;}

h1, h2, h3, h4, h5 { 

  margin: 2.75rem 0 1rem;

  font-family: 'Open Sans', sans-serif;

  font-weight: 300; /*correlates to Open Sans Light*/  

  line-height: 1.15;

}

h1 {

  margin-top: 0;

  font-size: 3.052rem;

}

h2 {font-size: 2.441rem;}

h3 {font-size: 1.953rem;}

h4 {font-size: 1.563rem;}

h5 {font-size: 1.25rem;}

small, .text_small {font-size: 0.8em;}

h2.Heading = 2.441em

h3.Heading = 1.953em

h4.Heading = 1.563em

h5.Heading = 1.25em

Accessibility

Note, in certain circumstances UL’s red brand color can present challenges to users with color blindness. At a minimum,

avoid combining red and green text/backgrounds/juxtapositions;

when using color to convey meaning (such as red text for error messages, green for success, etc.), also use symbols and words to convey that meaning.

In general, all text should be readable by measuring a contrast ratio of at least 4.5:1 (as low as 3.0 is acceptable in some circumstances) between text and background. A number of contrast checker online tools are available; Chrome developer tools also provides a tool to do this (tutorial).  Best practice is to make sure that the information is conveyed other than just by color.

Icons

Icons should be used as accents to support the topic and ideas being presented. As noted above, a combination of color, text and icons will convey the message best.

Use Font Awesome wherever possible for icons. Use the Light version, available to Pro users, when possible.

Font Awesome icons can be rendered as a vector image in any size of browser and don’t require any extra downloads or server resources; the library includes a large number of icons covering a wide gamut of objects.

On occasion, you may need to develop custom icons for special cases. Implement these icons in the code as SVGs, which allows for smooth and responsive resizing.

Responsiveness

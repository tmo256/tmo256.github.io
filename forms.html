Input types

Where appropriate, use HTML5 input types to ensure ease of use on touch devices. This includes, but is not limited to:

email
range
search
tel*
url
*Note: On mobile devices, tel type puts the user into the number keypad rather than the regular keyboard. Use tel as the preferred type when coding a numbers-only input.

Do not use the following types, as the implementation in some (or all) browsers may be confusing (“number” input, for example) or plugins implement the feature:

number
month
color
date
datetime
datetime-local
time
week

Buttons and Links

Forms commonly require actions other than a simple submit. In some cases, these actions should be performed by buttons and sometimes by links.

Note that although the rules below refer to buttons and links, the elements need not specifically be input, button or anchor tags; rather, we refer to the styling of the element (see “Anchors and Buttons” for more information on how to style these elements). For the vast majority of instances in the context of forms, however, links should be anchor tags and submit buttons input elements of type “submit” (see “Anchors and Buttons” for details on when to use each type of element).

Use the general guideline of the necessity of a URL to differentiate between using an anchor or a button.

Submit Events

As discussed above, although buttons and links do not necessarily need to correspond to their native HTML element tags, forms should generally be executed with input elements of type “submit.” Use a submit input instead of input of type=”button”. (See “Anchors and Buttons” for more information on when to use alternative element types.)

Any JavaScript in need of execution upon form submission should be attached to the form’s submit event, rather than attached to a click event of a button or link. Additionally, users will be able to submit forms by using the “Return/Enter” key, within a touch device keyboard, or by other methods (especially those used by accessibility browsers); attaching events to form submits circumvents all these issues.

JavaScript Validations

In most cases, JavaScript validations are not necessary. Some situations may make client-side validation more useful to the user, however.

Use client-side validations when:

the form is submitted via Ajax. Generally, this will make action flow more fluidly for the user.

the form contains a file upload input. Because of security issues, uploaded files are detached on page load, which makes round-trip validation poor UX (as the user is forced to reattach the file). (Note that some browsers expose more attributes of uploaded files than others, making specific validations relating to file size, type, etc., inconsistent across browsers.)

the form contains typeahead inputs that might be lost upon reload.

Best Practices

Implement the same error styling to the elements as the user would see from a server-side validation.

Validate on form on submit rather than on input. However, a multistep input, such as typeahead field, may be validated on value select for better interaction flow.

Clear errors on re-validation. Depending on the input type, as in the case of password field, clearing the error on field change may be more helpful to the user.

Success & Error Handling

As described in the “Alert” section, indicate success by including a visual alert in the resulting view. This alert should be dismissible (it is not necessary to programmatically close the alert after a set period of time).

When a view includes a form with error fields, an alert should be visible at the top of the view with a generic error message, such as “Please correct the input errors.” Do not list individual field errors in this message. Specify these errors by styling each invalid input.

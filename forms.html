In general, consider the following when creating forms:

  • As with all elements, practice accessible coding techniques with form fields, fieldsets and buttons.
  • Touch device behavior and display of inputs will impact your form, so consider your audience usage when designing. Mobile browsers may default to capitals, zoom unexpectedly, implement special field types differently and employ different fonts, among other considerations. Think about including attributes like autocapitalize, which only impacts virtual keyboards and voice-to-text input.
  • In some browsers, autocompletion of information, whether from past input or saved personal data, may affect field completion and validation. Consider field names and input attributes for each form field. For instance, you will likely suppress capitalization, spellcheck and autocomplete on search and typeahead search fields, but may wish to encourage autocompletion for contact information of the user's name and address.

Where appropriate, use HTML5 input types to ensure ease of use, particularly on touch devices. The email input type is especially useful in mobile browsers, but all types to consider include:

  • email
  • range
  • search
  • tel*
  • url
*Note: On mobile devices, tel type puts the user into the number keypad rather than the regular keyboard. Use tel as the preferred type when coding a numbers-only input.

Use the following types with caution, as the implementation in some (or all) browsers may be confusing and vary dramatically in presentation:

  • month
  • color
  • date
  • datetime
  • datetime-local
  • time
  • week
All major browsers support the number input, but display may vary. Because several browsers include some kind of spinner UI (up/down arrows to increment the input), some users may be confused by the type of field, while others are familiar with it. To mitigate this problem, you can use input attributes and browser-specific styling in your implementation, or consider an alternate input, such as tel.

Buttons and Anchors, Part II

Forms commonly require actions other than a simple submit. In some cases, these actions should be performed by buttons and in others, by links. Again, note that when we refer to buttons and links, the elements need not specifically be input, button or a tags; rather, we refer to the styling of the element (see "Anchors and Buttons" for more information on how to style). (Use the general guideline of the necessity of a URL to differentiate between using an anchor or a button. Can the user right-click or 3d touch on the link and open in a new window?)

Buttons
  • Submit (Create/Update)
  • Delete
  • Reset
  • Any action that initiates a data transaction
Links
  • Cancel
  • Forgot Password
  • Actions within tables (e.g. "Remove" link in a list, which performs a form submit)
  • Any other action that does not initiate a data transaction (but rather navigates to a different resource).

Submit Events

As discussed above, although buttons and links do not necessarily need to correspond to their native HTML element tags, forms should generally be executed with input elements of type="submit". Use this tag rather than <input type="button">.

Attach all JavaScript triggers to form and field events, however, rather than click events of a button or link. The allows users to submit forms or tab through form fields with keys, within a touch device keyboard, or by other methods (especially those used by accessibility browsers); attaching events to form submits circumvents problems that may arise from any alternate input scenario.

JavaScript Validations

In most cases, JavaScript validations are not necessary. Some situations may make client-side validation more useful to the user, however.

You may potentially want to use client-side validations when:

  • the form is submitted via AJAX. Generally, this will make action flow more fluidly for the user.
  • the form contains a file upload input. Because of security issues, uploaded files are detached on page load, which makes round-trip validation poor UX (as the user is forced to reattach the file). (Note that some browsers expose more attributes of uploaded files than others, making specific validations relating to file size, type, etc., inconsistent across browsers.)
  • the form contains typeahead inputs that might be lost upon reload.
  • moving from page to page in some kind of creation Wizard. Whether this pattern applies may depend on how the Wizard is implemented and dependencies between input fields from page to page.
  • looking for specific patterns on fields such passwords, where the text is obscured from view.

While client-side validation is often an expedient and helpful pattern, consider certain drawbacks:

  • Because resources may be much slower to load in mobile devices, users may begin filling out forms before JavaScript is applied to the form, leading to uncertain execution.
  • Is the validation executed on submit or on field change? Users commonly skip or half-complete fields for many reasons while filling out a form, and pre-emptive errors can lead to frustration for the user.
  • Browser autocomplete may act in unexpected ways, resulting in a false test outcomes.
  • If a form is long (or a viewport too small), inputs may be out of view on form submit, requiring a scripting solution.
  • As always, consider how JavaScript will interact with assistive technology devices and software; ensure the user gets a clear idea of what is happening by employing alert dialogs as described by WAI-ARIA best practices.
Best Practices
  • Do not put the user in "double-jeopardy" if at all avoidable. This means the user should rarely see a second set of errors after satisfying errors shown before form submit.
  • To avoid the above scenario, validate on form on submit rather than on input. However, a multistep input, such as typeahead field or password confirmation, may be validated on change for better interaction flow.
  • Implement the same error styling to the elements as the user would see from a server-side validation.
  • Clear errors on re-validation. Depending on the input type, as in the case of password field, it may be possible that clearing the error on field change may be more helpful to the user.

Success & Error Handling

As described in the "Alerts" section, indicate success by including a visual alert in the resulting view. This alert can be dismissible, though it is not necessary to programmatically close the alert after a set period of time.

When a view does include a form with error fields, an alert should be visible at the top of the view with a generic error message, such as "Please correct the input errors." Do not list individual field errors in this message. Specify these errors by styling each invalid input.

As always, consistent with the placement and styling of these types of messages. Use a combination of text, color and iconography to convey the message to the user.

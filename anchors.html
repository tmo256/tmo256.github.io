Anchors & Buttons

The guidelines below describe usage and styling of button and anchor elements outside the context of forms. See "Forms" for specific uses within that context. This section also does not address buttons as elements of an interactive environment, such as gaming environment (an expansion on the idea of interactivity in a more strictly "application" state will be included in future versions of this documentation).

Also, please note the recurring theme throughout this section: when using the terms "button" or "anchor," we are sometimes referring to the HTML button or a tags, and sometimes referring to a UI element that looks to the user to be something "pressable" or "clickable". We have made every possible attempt to be clear when referring to an HTML node and when referring to a UI element.

Buttons: Coding

Button WAI-ARIA Authoring

Use a button element to trigger any JavaScript function that does not use a URL. Basically, instead of using an a tag with an href="Javascript:void(0)", or href="#", use a button element. If the user can open a reference via a right-click or 3D touch, use an anchor tag instead.

Common button uses might include:

  • popover triggers.
  • tooltip elements.
  • modal triggers (that do not require a URL. Some modals that open with Ajax, for example, may be better served by an anchor tag with a URL in the href attribute).
  • dismissing buttons in modals and alerts.
  • opening and closing/expanding hidden text.

In many cases (such as some listed above), the button may appear to the user to be an inline link. Use CSS to modify the button to look like a link. (See "Styling" below for more information.)

Always use the type="button" attribute-value to ensure a button does not inadvertently act as a form submit. Safari, Chrome and other browsers treat button elements as submit tags when nested in forms.

Buttons: Styling

As noted, button elements may be styled to appear as either buttons or as links, depending on their conceptual use. Buttons create an impact within their environment, links navigate to resources.

Style elements as a button when:

  • indicating persistence or as a step in a process leading to persistence. This includes actions such as save, save and move forward or back ("next" and "previous"), submit, delete, update.
  • behaving as a component in an application; tools in a toolbar of a text editor, for example.
  • creating call to action (CTA). The number of CTAs should be limited on a view and should strongly indicate to the user what their next action should be.

When using icons to represent navigational elements, such as "hamburger" menu icons, the button node will most commonly be unstyled (without background or other skeuomorphic attributes). In most other cases (or unless specifically noted elsewhere in the document), style the button as a link in most other situations.

Buttons: Responsiveness

In most circumstances, elements styled as buttons should become block elements in medium and smaller (less than 768px wide) viewports in order to accommodate touch behavior; some exceptions to this rule might include "Next" and "Previous" buttons. Be sure to create adequate spacing and padding to allow all users to use your buttons on a touchscreen.

Buttons: Usability

  • Use sentence case for button labels.

  • Limit characters. Button text should be as short as possible with action words that clearly explain what the button does (Create Account, Save, Search).

  • Be consistent in this terminology across the application, such as "Save" vs "Submit".
  • Lead with a verb where possible. For example, "Create Account" not "Account Creation".

  • Icons can also be helpful. At times, consider adding an icon to signal specific actions (Download, Open in a new window, etc.).

  • Use consistent class variations to distinguish types and convey meaning. For example, "Previous" button steps should use a default or plain button class, while a submit or CTA button would implement a special primary.

Anchors: Coding

Whenever an element uses a URL or in-page navigation, use an anchor tag. This includes:

standard inline links.

jump nav links (href=#element-id).

AJAX modal triggers, where the user can also open content in a new tab (by right-click or 3D touch).

all cases described in "Forms".

when in doubt, prefer an anchor tag.

As noted above, a trigger for a modal may appear either as a link or a button, depending on circumstances. If the request is made using the URL in the href attribute, use an anchor.

When using an anchor tag as a button, include the role="button" attribute.

Anchors: Styling

Inline anchor style should use standard body font family and size, line height, etc, and, by default, an underline (text-decoration) to indicate a link to the user. This allows the designer to use colored text as a design element in headers or banners without creating confusion for the user; users should be able to identify links by more than color alone.

In some cases, the context of the link will provide the user with the indication of clickability; for instance, within a standard navigation in the header, or within a drop-down submenu. In those cases, underlines or other indicators may not be required.

Where an interactive link reveals (expands or creates) text, include an indicator of further content, such as an angle or caret icon. Ideally, this indicator would change orientation (animated) to indicate open and closed.

When designing for touch display, make sure to provide adequate space around every link, just as with buttons. Pay specific attention to lists and menus.

Input / Submits

Use styling described in the "Buttons: Styling" and "Buttons: Usability" sections.

Where possible, submit a form using input type="submit".

As noted in "Forms", attach any validation or pre-processing to the submit event for forms rather than click events on these elements. This will enable the use of the submit button, as well as ensure that submission by using the "Return" key or by assistive technology software will always execute the required JavaScript.

Other Cases

In addition to the cases noted above (primarily buttons that trigger JavaScript functions), buttons may be styled as links or vice versa.

All elements in an "Action" table cell should appear to the user to be links. The actions may be a mix of regular anchor tags, such as an anchor link to "Edit;" buttons, such as a confirmation modal triggered for a delete; or a submit button, as when submitting a form to make a saved search an alert.

Regular anchors may be styled as buttons for an extraordinary or analogous call to action, such as "Registration" links.

Where a mix of form submit and anchor elements should be viewed as siblings of equal importance, be consistent in displaying each element as either buttons or links.

Confirmation Dialogs

Use a confirmation dialog to confirm a process or persistence that cannot be easily undone or recreated; or, to confirm data persistence where the user may not be aware that persistence is about to occur (form inputs are not visible, for example). Most commonly, use this type of dialog when deleting or removing items. The dialog should interrupt the process to the point that the action cannot complete without explicit input (e.g., button click) from the user.

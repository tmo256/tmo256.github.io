Tables

Tables should be used to display easily scanned and/or comparable data.

Responsive Tables

Where tables are used in responsive design, ensure display in medium and small browser views by employing one or a combination of these methods:

Using data-labels attributes in table cell tags, combined with media query CSS to transform each cell into a stacked row. Many versions and tutorials on this method are available, including this one by Matt Smith. (See link for a visual example of this method.)

Pros: This method is widely supported and accommodates a lot of data legibly.

Cons: By stacking the data, comparing cells becomes more difficult.

Suppress less important columns where possible, allowing the remaining cells to fit comfortably in portrait mode on common mobile devices.

Pros: Supports the original goal of the table, keeping the contents scannable and comparable.

Cons: May eliminate useful information for the user. (In general, tables should not contain much information that is not necessary for the user. However, the circumstances under which the user uses a mobile device may change which pieces of information are more important.)

Add horizontal scrolling to the table’s parent object, while including some visual indicator that more content is available beyond the current view screen. Indicator might be a visible horizonal scrollbar or a gradient overlay.

Pros: Requires less coding and maintenance while displaying all the table data.

Cons: Horizontal scrolling is not always touch-device friendly; users have a hard time maintaining a connection between identifying cells. Creating the visual indicator of “more” can also be unclear.

Sorting

Prefer sorting by:

select/dropdown element.
table header ascending/descending.

Accessibility

Header cells should utilize the TH tag
All header cells should contain text.
Data cells should utilize the TD tag
